Q) What is Microservice
A) Microservice is a very small, autonomous application that is usually designed to perform one specific business functionality.
-> Microservice application is a very small and this is why it is called micro.
-> Microservice is a very small application that is responsible for one specific functionality only
-> It can be a small standalone application, or it can be a very small building block of a larger application.
-> But it is always loosely coupled so that if needed, it can be easily scaled horizontally.

Q) Microservice vs Monolithic application
Monolithic
----------
-> Monolithic is a Single Springboot Application with multiple controllers.
-> If we make any change in controller classes, then we have to build and redeploy the entire application

Microservice
------------
-> In Microservice, We have Multiple Smaller Applications.
-> Each Microservice is responsible for specific functionality.
-> Each Microservice works with its own Database. This is called Database per Service Pattern.
-> Making MS using its own DB, makes MS's Loosely Couple and More Scalable.

Q) Microservices Communication
A) One of the ways microservices can communicate with each other is by sending direct Http request and wait for Http response.
-> When it comes to direct communication, then you will come across synchronous and asynchronous Http requests.

Synchronous Communication (Synchronous HTTP Request)
-----------------------------------------------------
-> Example where 1 MS will wait until it receives HTTP Response from another MS, is called Synchronous Communication between 2 Microservices.

Asynchronous Communication (Asynchronous HTTP Request)
------------------------------------------------------
-> Order MS needs to notify Email notification microservice that a new order is created.
-> To do that, orders microservice will send http post request to email notification microservice API endpoint.
-> But in this case, order microservice will not wait for immediate response from email notification microservice.
-> It can send the request and immediately continue executing next job without waiting for email notification microservice to respond.
-> Even if the request was sent asynchronously, it is just response will come a little bit later.

Problem with Direct HTTP Request Communication
-----------------------------------------------
-> If email notification ms is temporarily not available even for 5 minutes, then it can loose many http requests.
-> If Any New MS is Introduced, and to start receiving notifications from Order MS, We need to Update Order MS to Send HTTP Request to New MS's.

-> And this is when Apache Kafka and event driven communication can help.

Q) Event Driven Microservices
A) We Use Apache Kafka to communicate between MS's.
-> Where 1 MS will publish the Event and Another MS will Consume the Event. This is called Publisher Subscriber Model.
-> As soon as Publisher publish the Event, All Subscribed microservices will receive this message.

Advantage
---------
-> In the case of direct request and response communication, if destination microservice is down, then most likely it will miss the messages and will
no longer receive it.
-> But in event driven architecture with Apache Kafka, if one of the subscribed microservices was down, it will still be able to consume events
that it missed when it comes back online and resumes working.

Q) What is Apache Kafka
A) Apache Kafka is distributed event streaming platform that is used to collect, process, store and integrate data at scale.
-> It is a platform that enables developers to build event driven applications that can continuously exchange data with each other, 
and we can scale this microservices as needed to support larger traffic.

Q) How Apache Kafka can be Used with Microservices
A) Apache Kafka can be used to help microservices communicate with each other.
-> Products MS will publish a product created event in the form of Json payload.
-> Now, because this product microservice can publish or it can produce an event in Kafka, it is called producer.
-> And to help us publish events, Kafka provides producer API.
-> So one of the core components in application that uses Kafka is producer.
-> Consumer is a microservice that consumes event.

Q) What is Kafka Cluster
A) A Kafka cluster is a group of multiple Kafka brokers working together. 
-> When we combined All Kafka Broker, they form a cluster that can handle data streaming with fault tolerance, scalability, and high availability.
-> Kafka Broker = a single Kafka server.
-> Kafka Cluster = multiple Kafka brokers working together.
-> Thats Why, to make your system always available and fault tolerant, you can start multiple brokers in a cluster.

Q) What is Kafka Broker
A) Kafka broker will handle all the work to accept messages from producer, store it reliably in topic partitions, and replicate it across other brokers
in the cluster.
-> It will also handle all the work to enable consumer microservices read events from topic partitions.
-> When a producer publishes data, data does not go directly from the producer to the consumer. Data 1st sent to Kafka broker. 
-> Broker will store Data in 1 of the Partitions. From there, broker makes the data available to consumers to consume the data.
-> Each Kafka Broker is a single Kafka Server.
-> Kafka Broker will replicate data across other brokers in the cluster.
-> To make our application more reliable, We will usually start more than one Kafka broker, and this way, if one Kafka broker is down, 
then another Kafka broker that is running can be used to service requests.
-> Among multiple Kafka brokers, only one broker is the controller (cluster manager).
-> For a topic with multiple partitions, each broker can be the leader for some of those partitions.

Hierarchy:- Kafka Broker -> Topics -> Partitions -> Offsets.
-> Kafka Broker stores Topics
-> Topics have multiple partitions
-> Partitions contains ordered messages. Each Identified by unique offset.

Q) What is Leader and Follower in Kafka Broker
A) Among Multiple Kafka Brokers, There will be 1 broker that will act as a leader, and all other brokers will act as followers.
-> Each of these brokers hosts Kafka topics, and each Kafka topic stores events in partitions.
-> This way, Kafka can achieve high availability and fault tolerance.
-> Now, the leader and the follower roles are not fixed to a single broker, but can change dynamically depending on the availability of the broker.
-> The leader will handle all read and write requests for the partitions, while followers will passively replicate leader's data.
-> If a Leader Kafka broker goes down for some reason, 1 of the follower brokers will become a new leader and will continue servicing requests.
-> Kafka broker is customizable and you can configure its behavior using configuration file.

Leader Kafka Broker
-------------------
-> A leader is responsible for handling all read and write requests for partitions in a topic. Manages all Operations for Topic Partitions.
-> Only Leader can perform Write Operations like storing Data in Topic Partitons.
-> So, Once message is successfully persisted in topic partition, it will be replicated to other Kafka brokers that act as followers.
-> Followers replicate data from the leader in exact order it was written, maintaining data consistency across cluster.
-> So leader is a single source of truth for all writes and reads in topic partition.
-> If followers were allowed to accept writes, it could lead to inconsistencies and conflicts because different followers might have different versions data.

Follower Kafka Broker
--------------------
-> A follower in Kafka is a replica of a partition. It copies all the data from the leader.
-> The main reason it is called a follower is because it follows a leader, meaning that it replicates whatever data the leader has.
-> Followers are very important for redundancy and fault tolerance.
-> If leader fails, one of the followers can take over.
-> If you have only one broker in the cluster and if that broker goes down, then your entire system is down and there is no replicas of your data.
-> So having multiple brokers in the cluster allows you to have more followers, and this helps you to ensure high availability of the data.

-> In Apache Kafka, Each partition will have a leader assigned to it, and a leader to a partition is assigned when the topic is created.
-> When you create a topic in Kafka, it also involves specifying the number of partitions for that topic.
-> Kafka, through its internal processes, assigns a leader for each partition right away, so each partition will have its own leader and followers. Example,
-> There is no single Kafka server that always acts as a leader.
-> if 1 Broker is down, Kafka will reassign leaders and followers to make sure that each partition has a healthy Kafka broker to work with as a new leader.

Partition	    Leader	    Follower 1	Follower 2
P-0	            Broker-1	Broker-2	Broker-3
P-1	            Broker-2	Broker-3	Broker-1
P-2	            Broker-3	Broker-1	Broker-2

Note:- Though Broker-2 and 3 are Followers of P-0, But Broker-2 is also Leader for P-1 and Broker-3 is also Leader for P-2 at the same time.

Q) Kafka Topic
A) Kafka Topic is durable storage of events information, inside the topic.
-> It is stored on a hard disk and it is replicated to other servers in Kafka cluster, meaning that you will have multiple copies of your data created.
-> This way if you lose one server, and if you lose all data that is stored on that server, you still have your data stored on other servers in Kafka cluster.
-> More specifically, Data is stored in a topic, which is managed by the broker.
-> The Kafka broker stores the data, and topics are the logical containers where the data (messages) are actually kept.
-> When event is consumed from Kafka topic, this event does not get deleted from topic right away.
-> It stays in the topic for as long as needed so that other microservices can also read this event.

-> Kafka Topics resides entirely on the Kafka Broker.
-> There is no concept of storing data in topic from Client end(Publisher). Client Can choose only Kafka Topic to send data.
-> Broker receives the messages and holds responsibility to store data in topics.

Q) Difference Between Kafka Message and Event
A) Consider Kafka message as a message envelope that carries something inside, and the event data are the content of this message envelope.
-> Event Data can be in different formats. It can be a simple string, Json payload or any other format or even null if needed.
-> Basically anything that can be serialized into an array of bytes you can use as a payload.

Q) Kafka Partitions
A) A Kafka topic is divided into multiple partitions.
-> When a publisher (producer) sends a message to a topic, the message is stored in exactly one partition of that topic - not across all partitions.
-> And Each Partition is replicated across multiple Kafka servers for durability.
-> In Which partition the message goes to, depends on the partitioning strategy:
-> If the producer specifies a key with the message, Kafka uses a hash function on the key to determine the partition. This ensures all messages with
the same key go to the same partition (helpful for message ordering per key).
-> If no key is provided, Kafka usually distributes messages round-robin or uses some other default partitioner to balance the load across partitions.
-> So, data is not duplicated across partitions; each message lands in one partition only.
-> This partitioning allows Kafka to scale horizontally and provides parallelism for consumers, because consumers can consume partitions independently.
-> Messages published to a topic are spread across partitions based on key or partitioner logic, but each message is stored in only one partition, 
not all partitions.
-> By default, when you create a Kafka topic without specifying the number of partitions, Kafka creates 1 partition.
-> Partitions have no fixed maximum size by default to store data in 1 partition. Instead, they are limited by retention policies (time-based or size-based).
-> So, partitions don't really "fill up" in the traditional sense, instead, old data gets deleted based on retention settings.
-> If we create a topic with more than one partition, we can horizontally scale our application and have a separate instance of the same microservice 
read from its own partition in parallel.

Example:- (Product-Created-Event-Topic)
---------------------------------------
Partition-0
Partition-1
partition-2

-> Consider SMS_Notification MS is running on 3 Different Servers and ready to consume messages as a consumer.
-> Now Once Event is Published from Product MS on "Product-Created-Event-Topic" topic, each SMS_Notification MS can read events parallelly from Partitions.
-> Like, 1st SMS_Notification MS reads from Partition-0, 2nd SMS_Notification MS reads from Partition-1 and so on...

Note:- You can increase the number of partitions in a Kafka topic to scale. But once set, partitions can’t be reduced.

Q) Kafka Partitions and Offset
A) Each Partition will have no. of Cells and Data is stored in this cells.
-> Each cell in a partition is numbered. This number is called offset. And this offset is very similar to index in Java array it starts with zero.
-> When a very first event is stored to partition zero, it will be stored with index zero.
-> The next event that will be stored to partition zero will be stored at index one, and every time a new event is stored to partition zero, 
its offset will increase by one.
-> Each Partition in Topic may have different no. of numbered Cells. Like Below Example,

Example:- (Product-Created-Event-Topic)
---------------------------------------
Partition-0 => 0,1,2,3,4..n
Partition-1 => 0,1,2
partition-2 => 0,1,2,3

-> So Kafka topic works on "append only" mechanism and a new event is always added at the end.
-> Now event data that is stored in a partition, it is immutable.
-> Once you store event to a partition, you cannot change it, you cannot delete it, and you cannot update it.
-> Event data that is stored in Kafka topic is immutable and that you cannot change it and you cannot delete it.

Scenario (1 Topic, 3 Partitions, 3 Brokers)
--------------------------------------------
-> Kafka will assign each partitions to a Leader from Kafka Broker.
-> Replication Factor = 1 (Each Partition exists on only 1 Broker). In this case, Partion-0 will get assign to only 1 Broker.
-> Each partition has a leader, which is a broker that handles all read and write requests for that partition.

Partition-0   Broker-1  (Broker-1 is leader for Partition-0. i.e. Broker-1 only can handle read and write requests to Partition-0)
Partition-1   Broker-2  (Broker-2 is leader for Partition-1. i.e. Broker-2 only can handle read and write requests to Partition-1)
Partition-2   Broker-3  (Broker-3 is leader for Partition-2. i.e. Broker-3 only can handle read and write requests to Partition-2)

-> When Broker-1 reads data from Partition-0, Kafka does not split data from Partition-0 into smaller partitions. It Will stored as-is.
-> Broker-2 and Broker-3 are not allowed to read or write operations from Partion-0 cos they are not the Leader
-> Producers: When they send data to Partition 0 -> they talk to Broker 1.
-> Consumers: When they read from Partition 0 -> they also talk to Broker 1
-> If Broker 1 goes down and replication is not enabled, which if we made replication factor = 1 only, then Partition 0 becomes unavailable since Broker
is down and No Other Brokers are enable to talk to partition-0. That's why replication is crucial for high availability.

-> With Replication Factor = 3, Broker-2 and Broker-3 have replica copies of Partition O.
-> If Broker-1 fails, Kafka will promote one of the replicas (Broker 2 or 3) to be the new leader.
-> This way, Partition 0 stays available

Q) How Long Data will get stored in Kafka Topic once Message is Consumed
A) After the message is consumed from Kafka topic, it remains there and does not get deleted.
-> It can stay there for as long as needed, but it does not stay there forever.
-> By default, Kafka Topic is configured to retain events for 7 days, but this value can change through configuration properties if needed.
-> You can change this value and you can make Kafka topics keep data for as long as you need it.

Q) What If Kafka Publishes Huge Data on Topic. Will Data divided and gets stored in across Partitions
A) Kafka does not automatically split a single large message across multiple partitions, no matter how large the message is.
-> Each message (record) published to a Kafka topic is stored in exactly one partition.
-> Even if the message is huge in size, it will not be split across multiple partitions.
-> The assignment of the message to a partition is based on:
a) The partition key (if provided), using a hashing algorithm.
b) Or, round-robin if no key is given and the producer does not specify a partition.

Q) Core Components of Apache Kafka
A) Publisher, Consumer, Kafka Broker, Topic, Partitions

Q) Ordering of Events in Apache Kafka.
A) So let's assume we have a topic "Profile_Update" Topic, that is called profile updated event topic.
-> Every time user updates their profile, a new event is published to this topic.
-> When message key is not provided, then this event can be stored in any partition. eg:- Plain String => Sergei
-> When message key is not provided, Kafka will try to load balance events and store a new event in one of the available partitions.
-> Let's assume that user changed the spelling of their first name and clicked on the update button.
-> A new event is published and it is persisted into Partition-0
-> With this change, the spelling of user's first name has letter I at the end.
-> Then this user changed their name again and clicked on the update button again.
-> A new event is published and this time it was stored in a Partition-1 and again new Event if name is changed again and stored in Partition-2.

Example
--------
Event-A:- Partition-0 => Sergei
Event-B:- Partition-1 => Serjei 
Event-C:- Partition-2 => Sergey

Scenario-1
----------
-> Consumer microservice will consume events from topic partitions in parallel.
-> And because Kafka consumer reads events in parallel, there is no guarantee in what order consumer microservice will receive these events.
-> It could read event A first, then C, then B, it could read events in order B, A and C, or it could read events in order C, B and A.
-> And this can be a problem if the order in which these events should be processed is important.
-> The order in which we want these events to be processed is A, B, and C, and this is very important, because if the order is not followed, 
we will update user's first name in the database with incorrect value.
-> So in order to read the Events in the order which events got published, we should always provide message key.
-> If message key is provided, then Kafka will use it to determine which partition this event should be written to in Kafka topic.
-> When the same message key is used for different events, those events will be stored in the same topic Partition.
-> All these 3 Events will get stored in same Partition, since message key will be same and saved end of the cell in each partition for every new Event, like
below example

Example
--------
Key-Value => firstName:Sergei
Event-A:- Partition-0 => Sergei
Event-B:- Partition-0 => Serjei
Event-C:- Partition-0 => Sergey

-> It will take message key, it will hash it, and then it will use this hash value to determine which partition to use to store this event.
-> And it always stores messages with the same message key in the same partition. And this is how Kafka helps us to achieve ordering of events.
-> In this case, when consumer microservice reads events from topic partitions, even if it reads events from other partitions in parallel, 
the order in which events with the same message key is published will get processed in the same order which event got published.

Note:- Kafka ensure message ordering across partitions By using a key for each message and partitioning by the hash of the key.

Scenario-2 (1 Topic, 1 Kafka Broker, 3 Partitions, 1 Consumer)
--------------------------------------------------------------
-> 1 Consumer can read data from all partitions. Irrespective in which partition data will get store, Consumer will read data from across partitions.
-> 1 Consumer can read data from multiple partitions in a non-sequential manner.
-> When Consumer is reading from Partition-0, It may read Partition-1 as well before completing Partition-0.
-> Consumer Won't finish 1 partition completely before moving to next partition.
-> But Within each Partition, Messages are consumed in order (based on their offsets).

Case-1:- Does Consumer read partition in sequence      =>  No. It reads from all partitions but not in fixed order.
Case-2;- Does Consumer read all assigned partitions    =>  Yes
Case-3:- Is Order Guarantee across partitions          =>  No. Only Within a partition.


Q) How and when it get decided that published data will get stored in 1 partition or distributed across partitions
Scenario-1:- If Explicit partition Provided                =>    Message goes to that partition only.
Scenario-2:- If Key Provided and no partition mentioned    =>    partition chosen by hashing the key.
Scenario-3:- No Key, No Partition specified                =>    Messages Distributed evenly (Round Robin).

-> So, When all data is stored in single Partition, Kafka guarantees order of messages.
-> But When Data is Spread across multiple partitions, Kafka Does not guarantee the order across those partitions.
-> So, Partition Selection is typically handled by the Producer and Not Broker.

1) Role of a Broker
2) Role of a Controller

Q) Stop Apache Kafka Servers or Broker
A) But Before shutting down your servers, it is a good practice to stop all running producers and consumers.
-> This will also help you avoid losing messages because if you stop your servers, they are down, but your Kafka producers keep on sending messages
to those offline servers. There is a chance that you lose those messages.
-> Stopping producers and consumers first also helps you to avoid errors in your system.
-> This is because if your producers and consumers keep on interacting with the server that is going down, this will cause exceptions.
-> A Better way to stop one or multiple Kafka servers gracefully is to use CLI script.

Log segments are closed and any necessary cleanup is performed.
-> By executing cmd "./bin/kafka-server-stop.sh" in new Terminal. It will gracefully shutdown all 3 Kafka Servers.

Consume Kafka Messages in Order from Kafka Topic
------------------------------------------------
-> If you remember from previous video lessons, Kafka Topic is actually split into multiple partitions.
-> To determine which partition to choose to store a message, Kafka will use a hash of a message key.
-> If there is no key, then Kafka will equally distribute messages across multiple partitions using round robin algorithm.
-> And if you want to read messages in the order they were stored, these messages need to use exactly the same key.
-> If messages have the same key, they will be stored in the same partition and they will be read in the same order.
-> Will create a new Topic with "messages-order" Topic name by executing below command,

-> Now Publish Key Value pair data in Topic by executing below command,

>1:Second Message
>1:third message
>a:a
>b:b
>c:c

-> Now Open new tab in terminal for consumer to consume messages from beginning, execute below cmd from bin folder,
./kafka-console-consumer.sh --topic messages-order --bootstrap-server localhost:9092 --from-beginning --property print.key=true (Enter)
1  First message
1  Second Message
1  third message
b  b
a  a
c  c

-> Above is the messages will display which has been consumed by consumer.
-> Now notice that a list of messages with the same key. They are all ordered.
->And those messages that were stored with a different key, Even though we send them in order, their order is different when we read them.
-> This is because if message has a key, then its partition will be determined based on the hash of that key.
-> If messages have the same key, they will go into the same partition and thus will be read in the same order.
-> If messages are stored without a key, then technically the key is null, and in this case, Kafka will distribute these messages among multiple 
partitions using a round robin algorithm.
-> And even if you send these messages in order, when reading them, their order will be different.

Note:- 2 consumers from the same consumer group cannot read messages from the same partition.

Q) Publishing Data over Kafka Topic Synchronously
A) If I send Kafka message synchronously, then I can configure my microservice application to wait for acknowledgement from all Kafka brokers that 
my message is successfully stored in Kafka topic.
-> When message is sent asynchronously, we do not have the guarantee, and it is possible that we lose this message if an error takes place.
-> Add below line of code, to make our Kafka Synchronous
future.join();  //Synchronous code. Only 1 line Change from Asynchronous code.

-> With this line of code, even though the Completablefuture is used, addition of this join method makes this code effectively synchronous, 
and this means that the thread will wait at this line until the Completablefuture completes.
-> if we add "future.join();" then your code is synchronous and if you do not have this line, then your code is asynchronous.

-> But the disadvantage of this approach is that it can easily confuse other developers and make them think that this code is asynchronous, 
and this is because it contains keywords like completablefuture and the variable is called future as well.
-> Developer reading this code might not notice you calling join at the bottom of this method, and might think that this code is asynchronous.
-> So another way to write this code so that it sends message synchronously is to remove below lines from code,
1) future.join();
2) whole future.whenComplete(..)...
3) Remove CompletableFuture from CompletableFuture<SendResult...
4) Change variable from future to result.
5) Add .get() at the end of the kafkaTemplate.send(..) method.

-> Advantage of sending message synchronously is that we can configure our microservice to wait for acknowledgement from all Kafka brokers that the 
message is successfully stored in Kafka topic
-> But because the send method will now wait for a response, the execution will be just a little bit slower.
-> But at least we have a guarantee that our message is not lost, and it is indeed persisted in Kafka topic.

-> Final Code for Synchronous will look like below,
SendResult<String, ProductCreatedEvent> result = kafkaTemplate.send(productCreatedtopicName, productId, productCreatedEvent).get();
return productId;

Q) Kafka Producer Acknowledgement
A) TO Enable Kafka Producer Acknowledgement, Below is the Property
spring.kafka.producer.acks=1   //1 is Default Value
-> When Kafka producer sends a message, this message is sent to Kafka Broker.
-> Kafka broker stores this message in Kafka topic.
-> Inside the topic, Kafka message is stored in one of the partitions.

-> Now, if your Kafka cluster has more than one broker and proper replication factor configured, then topic partitions will be copied over to other 
brokers as well.
-> In this case, one broker is a leader and it is the first one to receive Kafka message.
-> It receives message and then it copies this message over to other brokers that act as followers.
-> And these brokers that act as followers, They simply replicate messages of the leader broker by default.
-> Once leader broker stores message successfully, it sends acknowledgement to Kafka Producer that all is good, that it was able to successfully 
store the message and Kafka producer by default, it is configured to receive this acknowledgement from one broker only i.e from leader.
-> It is very useful configuration because it confirms that the received message is stored.

Q) What if Leader Broker goes down before sharing this message with its followers and never comes back?
A) In this case, message is lost. To make our system more reliable, we can configure Kafka Producer to wait for acknowledgment from other brokers as well.
-> We should configure Kafka Producer to expect acknowledgment from more than one broker, not only from the leader, but also from all followers.
-> This will make our system work just a little bit slower, because in this case, Kafka producer will wait for acknowledgment from a leader and 
it will wait for acknowledgment from all followers.
-> And in this case, when it is critical to ensure that the message does not get lost, you will want to configure your Kafka producer to wait for 
acknowledgment from all in-sync replicas.
-> And in Case, if message is not very critical and Okay if lose some messages, you can configure your producer not to wait for any acknowledgment at all.

Scenario-1 (Kafka Producer Will Wait for acknowledgement from all Brokers)
---------------------------------------------------------------------------
-> To configure your Kafka producer to wait for acknowledgement from all brokers, below property need to be configured,
spring.kafka.producer.acks=all
-> In this case, Kafka producer will wait for a full set of in-sync replicas to acknowledge the record.
-> This option provides the strongest durability guarantees because no data will be lost as long as at least one in-sync replica is alive.

Scenario-2 (Kafka Producer Will Wait for acknowledgement from Leader Broker Only)
---------------------------------------------------------------------------------
-> To configure a Kafka producer to wait for acknowledgment from a leader broker, below property need to be configured,
spring.kafka.producer.acks=1
-> It ensures that Kafka message has been received and stored by the leader broker, but it does not guarantee that Kafka message has been replicated to follower brokers.
-> Some data may be lost if leader broker fails before followers catch up, so this option is faster than waiting for all brokers to acknowledge, but it is less reliable.

Scenario-3 (Kafka Producer Will Not to Wait for any acknowledgement at all)
----------------------------------------------------------------------------
-> To configure your Kafka producer not to wait for acknowledgements at all, below property need to be configured,
spring.kafka.producer.acks=0
->This might be helpful in those cases when your Kafka producer sends a lot of messages and none of those messages are very critical.
-> And because Kafka Producer does not wait, your system works faster, but there is no guarantee that all messages will be stored.

Q) What is Batch and Linger
A) Below is a Kafka producer configuration that controls the linger time for batching records before sending them to the Kafka Broker.
spring.kafka.producer.properties.linger.ms=0
-> If linger.ms=10 ms:
• When the 1st message arrives at time 0 ms, the producer starts a timer for 10 ms (because linger.ms=10 ).
• If any other messages arrive within those 10 ms, like the 2nd message arrived at 5 ms, they are added to the same batch.
• At the 10 ms mark, the producer sends all the messages collected so far (both the 0 ms and 5 ms messages) together in 1 batch.
-> If linger.ms=0 ms:
• The producer sends the first message immediately (at 0 ms).
• When the 2nd message arrives at 5 ms, it sends that immediately too. It wont wait for 5 ms. 5ms is just the arrival time of the 2nd message.
-> Below config, configures the maximum time producer can wait after sending a single request.
spring.kafka.producer.properties.request.timeout.ms=1

Q) Difference between delivery.timeout.ms and request.timeout.ms
spring.kafka.producer.properties.delivery.timeout.ms=120000 //Default Value is 2 Minutes.
-> This 2 mins timeout includes
1) Time the messages sits in the batch waiting for linger.ms to expire
2) Retries
3) Time Spent waiting for acknowledgements (affected by acks and request.timeout.ms)

spring.kafka.producer.properties.request.timeout.ms=30000
-> This is the maximum amount of time the Kafka client (producer) will wait for the broker's response after sending a produce request.
-> Setting it to 1 (ms) means the producer only waits 1 millisecond for the response from the broker.

• delivery.timeout.ms ≥ request.timeout.ms + (retries x backoff)
• request.timeout.ms is about broker response wait time. //set to high cos if broker goes down, atleast it will take more than 1ms to restart
• delivery.timeout.ms is about overall send attempt.

Q) Idempotent Kafka Producer
A) Once you enable Idempotence in your Kafka producer, your application will work this way.
-> Producer will send a message.
-> Kafka broker will store this message in Kafka topic and once the message is stored, Kafka Broker will send acknowledgement back to producer.
-> If network problem takes place and this acknowledgement does not reach Kafka producer, then our producer will retry and send the message again.
-> But because we have enabled Idempotence in our producer, Kafka broker will see that producer is sending the same message again.
-> Since, there is already a message like this in Kafka topic. Instead of storing the same message again, Kafka Broker will simply acknowledge that it does have already this message and this time acknowledgement receives by broker.
-> And in this case, if Kafka producer sends the same message multiple times, there will be only one such message in the topic and there will be no duplicates.
-> In simple words, Producer can send the same message multiple times, but Kafka Broker will only store it once and keep it in the correct order, and this can prevent inconsistency in Kafka topic, even if there are problems with the network or servers.

Q) Enable Idempotent Kafka Producer
A) To enable Idempotents in your Kafka producer, you will need to set the following configuration property to true.
-> If we are using application.properties file, then below is the property
spring.kafka.producer.properties.enable.idempotence=true

-> If you're using bean method, then your configuration property will look like this.
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG,true);

-> By Default, Idempotent configuration property is set to true and you do not really need to enable it explicitly.
-> But it is recommended to set property to true.

Conditions to Enable Idempotent Kafka Producer
----------------------------------------------
1) spring.kafka.produce.properties.acks=all
-> If you are using application.properties file, then for Idempotence to work, acknowledgment property will need to be set to "all"
2) spring.kafka.produce.properties.retries=2147483647  //2147483647 is by default
-> All retries must be > 0
3) spring.kafka.producer.properties.max.in.flight.requests.per.connection=5
-> maximum in-flight requests per connection must be <= 5.
-> By Above Property means, Kafka producer can send up to five requests or batches of messages to broker at the same time without waiting for acknowledgements, and this can improve throughput and performance of your broker.
-> It means, 1st msg sent to broker, and didnt receive acknowledgement from Broker to client. But he can send 2nd message as well upto 5 messages.

Kafka Consumer
--------------
-> When consumer starts reading messages, it starts pulling messages from Kafka topic at a regular time interval.
-> When Kafka consumer reads messages from partitions, it reads them in parallel.
-> There is no order guarantee which messages from which partitions will be read first, but it does read messages in order within a single partition.
-> So if we take partition one, for example, then the message from offset 0 will always be read before message with offset 1, Next it will read message from offset 2, and next it will read message from offset 3.
-> So messages within a single partition, They are always read in order, but there is no order guarantee between partitions.

-> For example, there is no guarantee that messages from partition 1 will always be read messages before from partition 2, there is no order in which messages are read between partitions.
-> If you have a single application, then it will read messages from all three partitions.
-> But if you have 3 consumers running, then each consumer will be assigned to read messages from 1 partition only.
-> If we scale and start more than 1 instance of the same consumer MS, then all instances of Kafka consumer can be grouped together and work as a group.
-> This way, instead of 1 consumer, you will have 3 consumers reading messages from the same topic, each consumer reading messages from its own partition.
-> And this will help you process messages from Kafka topic faster.

-> Once Kafka consumer reads message from Kafka topic, it does not delete this message from the topic.
-> The message remains in the topic until it is deleted from there automatically.
-> By default, Kafka Topic is configured to keep messages for 168 hours, which is 7 days, but if needed, you can change this value using Configuration properties file.

Q) Consume Messages or Events By Java Code
A) To Consume Messages, will use @KafkaListener on top of method.

-> When we apply this annotation on top of method, then it means that this method should be invoked whenever a new message is received from specified Kafka topic and the topic name you will provide in brackets. below is the example,

-> To specify which particular event your method is listening to, you will pass event as a method argument. in our case, we are publishing "ProductCreatedEvent" event. so to consume same message, will use "ProductCreatedEvent" data type.

@Component

@KafkaListener(topics="product-created-events-topic")

	@Value("${product-kafka-topic}")
    String topicName;
	
	@KafkaHandler
	public void handleProductCreated(ProductCreatedEvent productCreatedEvent) {
		log.info("Consumed Messages from Topic:{}",topicName);
		log.info("Messages are:{}",productCreatedEvent);
	}
}

-> In Above Code, If a message payload can be deserialized to ProductCreatedEvent, it goes to handleProductCreated, which is what in our case.
-> In case, If payload matches with OrderCreatedEvent, it goes to handleOrderCreated.
-> If it’s just a string, it goes to handleString.
-> This is a powerful feature for handling heterogeneous message types from a single topic in Spring Kafka.
-> 1 Consumer can consume messages from Multiple Kafka Topics.
-> No. of @KafkaListener for each topic defines the no. of Consumers we are using.

-> we can look at these messages later and decide what we want to do with them later. You can inspect messages that are in the dead letter topic.
-> You can identify the root cause of their problem, take corrective actions, and maybe even create a separate consumer microservice that will consume messages from dead letter topic and process them differently.

Q) Kafka Consumer - Multiple Consumer Groups
A) Let's assume we have 3 Consumer MS. Deposit, SMS Notification and Push notification microservices.
-> Deposit MS is running on 4 Instances, Where as SMS and Push are running on 1 Instance each.
-> In this case, when a new message is published to a topic, a SMS notification microservice will receive its own copy and push notification microservice will receive its own copy.
-> But how about Deposit microservices ? We dont want same message will be processed by Deposit microservice 4 times since its running on 4 instances.
-> Even though we have 4 instances of Deposit MS running, we want only 1 of them to receive a new message, and it does not matter which 1 of these instances will process the message. We want the message to be processed 1 time only.
-> So to Scale up consumer microservice, Kafka allows us to run multiple consumers in a group, and this is called consumer group.
-> Running multiple consumer microservices in a group is very helpful when you have lots of messages in the topic, and you need to process them faster.
-> In this case, Up multiple instances of Deposit MS, group them into 1 consumer group, and have each group member pick 1 message from the topic.
-> And in this case, messages that are stored in the topic will be processed much faster because instead of 1 microservice, you now have multiple copies of this microservice working.
-> But even though you have multiple instances of the same microservice working, when a new message gets into the topic, it will be processed 1 time only by 1 of these Deposit MS instances in the consumer group.

Q) How Group-ID Works in Kafka Consumer
1. Deposit MS Running on 2 Instances
-------------------------------------
-> If Deposit is running on 2 Instances then we can define group-id. 
-> Since 2 Instances are running on same MS, Both instances of Deposit MS must use the same group-id. 
-> You do not change the group-id between instances. it must be the same for all instances of Deposit MS that are working together.
-> Kafka will automatically distribute partitions of the topic between these two instances.
-> Each instance will consume messages from its assigned partitions only.

2. Deposit and Withdrawal MS (Different Consumer Microservices)
----------------------------------------------------------------
-> Deposit and Withdrawal are 2 Different Consumer MS.
-> If you run 2 instances of Deposit MS, both instances uses the same group-id.
-> If you run 2 instances of Withdrawal MS, both instances uses the same group-id (but different from Deposit’s).
-> If 2 instances of same MS then group-id should be same, But Group-id will be different for Deposit and Withdrawal.

Q) Rebalancing and Partition Assignment in Apache Kafka
A) Till now we have worked on 3 Kafka Brokers, 1 Producer, 1 Topic with 3 Partitions and 1 Consumer. Topic Name:- product-created-events-topic
-> All messages in this Kafka topic will be stored in this 3 separate partitions.
-> When we have only 1 consumer microservice, then this consumer will read messages from all 3 partitions in a topic.
-> Now We want to scale up our application, and we want to start up 1 more Consumer MS instance. Now we have 2 Consumer MS in Total.
-> What will happen now is, Kafka will notice that we have 2 consumer applications running, and it will reassign partitions among these 2 consumers.
-> Now it is not 1 consumer ms that reads from all 2 partitions.
-> 1 instance of consumer ms reads messages from 1 partition and another instance of consumer ms reads data from other or 2nd partitions.
-> This process of automatic partition reassignment is called "Rebalancing"
-> It happens very quickly, but while partitions are being reassigned, consumers stop consuming messages from topic for a moment.
-> And rebalancing takes place every time a new consumer joins a consumer group, and every time existing consumer leaves consumer group.

-> If 1 more consumer is UP as 3rd consumer, then partition rebalancing will automatically take place again and partitions will be reassigned again.
-> And during this partition reassignment, Kafka is trying to make sure that each partition is consumed by only 1 consumer in a group.
-> Once Rebalancing is complete, all consumers in the group will continue to consume messages from their assigned partitions in parallel.

-> If we scale and make 1 more consumer ms UP, Total 4 Consumers now. Unfortunately, this instance will sit idle, since we have 3 partitions only.
-> 2 consumers from the same consumer group cannot read messages from the same partition.
-> You cannot start more consumers than the number of partitions you have in the topic.
-> If your topic has only one partition, then you can start only one consumer microservice.
-> If you start more consumer instances than no. of partitions in your Kafka topic, then those extra instances will sit idle.

-> Now when you start a new Kafka consumer application and it joins consumer group, this new Kafka consumer application starts sending signals to one of the brokers that is selected as a group coordinator.
-> These signals, they are called "Heartbeats" and they are sent at irregular time intervals.
-> This "Time interval" can be configured, but the default value is 3 seconds.
-> So while your Kafka consumer ms are running, each of them will be sending heartbeat signals to Kafka Broker, letting Kafka Broker know that I'm running and I can consume new messages.
-> If 1 of these Kafka consumers stops, then it also stops sending heartbeat signals to Kafka Broker.
-> When Kafka Broker notices that there are no heartbeats coming from consumer, it will remove this consumer from consumer group and it will initiate Rebalancing.
-> It will check how many consumers are running and how many partitions you have in Kafka topic, and it will reassign partitions, trying to make sure that each partition is read by 1 consumer only.
-> Remember, 2 consumers from the same consumer group cannot read messages from the same partition, and when rebalancing completes, all consumers in the group will resume consuming messages from newly assigned partitions in parallel.

Q) What is Idempotent Consumer
A) If Kafka producer sends the same message multiple times, idempotent consumer will process it only once.
-> So Kafka Consumer will consume a message, it will start processing this message.
-> While processing all these messages and doing all business operations, it is possible that this might takes a very long time to complete, and because it took a longer time to complete, it exceeded the time configured with Property "max.poll.interval.ms" in milliseconds.
-> Now this configuration property, it controls how much time Kafka Consumer has to process full messages before it should pull a new batch of messages from Kafka topic.
-> If Kafka Consumer does not pull new messages from Kafka Broker longer than the time configured with this property, then Kafka Broker will assume that this consumer has failed, or it stopped and it will remove it from consumer group, and then it will trigger rebalance to reassign partitions to other consumers
-> so that other instance of Kafka consumer can consume and process this message.

-> If this happens, then 1st instance of consumer microservice which failed to read data, it will not be able to update partition offset to Broker.
-> So Kafka Broker will deliver the same msg to another instance of the same ms, or even to the same instance when this instance rejoins the group.
-> But either way, the same message will be consumed again, and then Kafka Consumer will start processing the same message for the 2nd time.
-> This time it will be able to complete all the business logic in time and it will update partition offset to Broker saying this message is consumed successfully.
-> But look what happened. We processed only one message, but there are 2 records in the database and we published 2 messages to another topic.

-> If Kafka producer is not idempotent, it can produce duplicate messages.
-> So your Kafka consumer should be able to avoid processing the same message multiple times.
-> And 1 of the steps on the way to avoid situations like this is to make your Kafka consumer idempotent, meaning that it can receive the same message multiple times but it will process it only 1 time.

-> Ideally, to avoid duplicate messages in your application, you will use multiple techniques at the same time, and idempotent consumer is only part of the solution.
-> Using idempotent consumer does help to avoid processing the same Kafka message twice, but it does not help inn all cases.
-> There will still be edge cases when Idempotent Consumer alone is not enough.
-> And another technique we can use to avoid processing the same message twice is to make your Kafka producer idempotent as well.
-> Because that Idempotent producer helps you to prevent sending duplicate messages to Kafka.
-> Consumer making producer and consumer idempotent is a good combination.

Avoiding Duplicate Messages, Use Below techniques
1) Idempotent Kafka Producer
2) Idempotent Kafka Consumer
3) Transaction

How Exactly Idempotent Works in Kafka Consumer MS
--------------------------------------------------
-> Once we Publish Data with messageID passed in Header, Kafka Broker Stores this Data in Topic.
-> Consumers will read same data from Kafka Brokers.
-> If a consumer fails to read messages or leaves the group, Kafka will rebalance partitions among remaining consumers.
-> Kafka Broker Will not send the msgs again rather, partition ownership changes, and new consumer will read uncommitted messages from the partition.
-> Once we publish, we are generating messageID and this messageID gets stored in Topic By Kafka Broker.
-> Since Broker has copy of messageID, new consumer will read uncommitted messages from the partition in case any consumer fails to read in 1st Attempt.
-> Since we are generating messageID with UUID, If something goes wrong from consumer end to read data, Its not like, messageID will be generated again.
-> In Consumer By Logic, will check if messageID is present. based on true or false we are making sure Idempotent is working.

Q) How Apache Kafka Works if we enable Transactions
A) Transactions help us write to multiple topics atomically and transactions help us achieve all or nothing behavior.
-> If we need to write to multiple topics, then it is either all writes are successful or none of them are successful.

-> @Transactional annotation is provided by Spring Framework. It is not specific to Apache Kafka.
-> When u use @Transactional on top of method, you tell Spring Framework that code in this method should be executed within 1 transaction.
-> All operations in this method that support transactions, they should be treated as a single unit of work, meaning that they should either all succeed or none of them should succeed.
-> If the method completes successfully, then transaction commits. Otherwise, transaction rolls back.

Important Notes on @Transactional
----------------------------------
-> @Transactional without any parameters is a Default Transaction Manager.
-> Spring will use the default transaction manager configured in your application context.
-> If you have a single Datasource, Spring Boot will auto-configure usually named transactionManager.

-> @Transactional("transactionManager"), When you use @Transactional with a value is Explicit Transaction Manager.
-> You are explicitly specifying which transaction manager bean to use (by its name).
-> This is useful when you have multiple DataSources and you want to control which one is used for a particular method or class.

-> If i have Single DataSource and if i use @Transactional("transactionManager"), will behave same like @Transactional though i have explicitly defined.

Q) Enable Kafka Transaction in Kafka Producer using Props File
A) From Now on, Transfer Microservice will be our Producer and "Withdrawal and Deposit Microservice" will be our Consumers.
-> To Enable Transaction in Producer, we have configured below property in props file
spring.kafka.producer.transaction-id-prefix=transfer-service-${random.value}-  //${random.value} is dynamic like transfer-service-12345
-> Each producer instance will have a unique transactional ID like transfer-service-12345-0, transfer-service-12345-1, etc.
-> Transactional IDs uniquely identify a producer’s transactions. 
-> Kafka uses these IDs to track and manage transactions, especially across restarts or failures.
-> Incase if my application crashes in the middle of transaction, Kafka can use this transaction ID to identify which messages were part of incomplete transaction and make sure that these messages are not processed until transaction is complete.

Note:- Above Property is not enough to enable Kafka transactions by itself. This property only enables transactional capability for your Kafka producer.
-> You must use @Transactional on the method where you send Kafka messages.
-> This ensures that all Kafka send operations within the method are executed within a single Kafka transaction. 
-> If an exception occurs within the transactional method, none of the messages will be committed to Kafka.
-> The transaction will be rolled back, and consumers will not see any of the messages sent within that transaction.

Q) Kafka Transactions: End-to-End Flow
Producer Side
--------------
-> The producer is configured with a transactional ID (from the above property).
-> The producer starts a transaction (beginTransaction()).
-> The producer sends multiple messages to one or more topics/partitions as part of this transaction.
-> The producer commits (commitTransaction()) or aborts (abortTransaction()) the transaction.
-> If the application crashes before committing: The transaction is left in an "incomplete" or "open" state.

Broker Side
-----------
-> The Kafka broker keeps track of all ongoing transactions using the transactional ID.
-> If a producer with the same transactional ID reconnects (e.g., after a crash), the broker knows if there was an incomplete transaction.
-> The broker ensures that:
1) Messages from incomplete transactions are not visible to consumers.
2) Only messages from committed transactions are delivered to consumers.

Consumer Side
-------------
-> Consumers that are configured for read_committed isolation (the default in Spring Kafka when using transactions) will only see messages from committed transactions.
-> Messages from incomplete or aborted transactions are not delivered to consumers.

Q) What is @Transactional("kafkaTransactionManager")
A) For Kafka, Spring Kafka provides a transaction manager called KafkaTransactionManager.
-> If you define a bean named kafkaTransactionManager in your Spring context, you can use @Transactional("kafkaTransactionManager") to indicate that the method should run within a Kafka transaction.

When to Use
-----------
-> Use @Transactional("kafkaTransactionManager") when you want to ensure that your Kafka operations (and any other transactional resources managed by this transaction manager) are executed within a Kafka transaction.
-> This is especially useful if you have multiple transaction managers (for example, one for your DB and 1 for Kafka).

Important Notes
----------------
-> Kafka transactions are only supported when using a transactional producer (transactional.id property set and transactions enabled in your Kafka configuration).
-> You must ensure that your KafkaTemplate and producer factory are properly configured for transactions.

Note:-
------
-> @Transactional("kafkaTransactionManager") will manage only Kafka related Transactions whereas
-> @Transactional("transactionManager") will manage both DB and Kafka related Transactions. Certainly All Transactions.

Q) If We already made Producer as Synchronized Transaction then what is the use of "READ_COMMITTED" property in Consumer MS
A) Setting "spring.kafka.producer.transaction-id-prefix" makes your Kafka producer transactional.
-> By Above Property, Producer is only going to publish Committed msgs only.
-> But there are scenario where we also need to configure our consumer to read committed msgs only.

Scenario 1: Transactional Producer, Consumer with Default Isolation Level (read_uncommitted)
---------------------------------------------------------------------------------------------
Flow:
-> Producer publishes (with transactions enabled):
-> Producer sends messages to the Kafka topic (these messages are not yet committed).
-> Consumer configured to read uncommitted messages as well (isolation level not set, so defaults to read_uncommitted):
-> Consumer can see and process these messages even though the transaction is not yet committed.
-> Exception arises (in producer or in the transaction):
-> Producer rolls back (aborts) the transaction with "UNCOMMITTED"
-> Messages are discarded by Kafka and will not be visible to consumers with read_committed.
-> But consumers with read_uncommitted have already seen and possibly processed these messages!

Key Point:
----------
With read_uncommitted, consumers can read messages that are part of transactions that are later aborted. This can lead to data inconsistencies.

Scenario 2: Transactional Producer, Consumer with Isolation Level (read_committed)
-----------------------------------------------------------------------------------
Flow:
-> Producer publishes (with transactions enabled):
-> Producer sends messages to the Kafka topic (not yet committed).
-> Consumer configured to read only committed messages (isolation.level=read_committed):
-> Consumer cannot see these messages until the transaction is committed.
-> Exception arises (in producer or in the transaction):
-> Producer rolls back (aborts) the transaction with "UNCOMMITTED"
-> Messages are never visible to the consumer since consumers only allowed to see "COMMITTED" msgs only
-> No risk of processing uncommitted or aborted messages.

@Configuration
public class KafkaDepositConsumerConfig {
	
	@Autowired
	Environment environment;

	@Bean
	ConsumerFactory<String, DepositRequestedEvent> consumerFactory() {
		// Configure the JsonDeserializer
	    JsonDeserializer<DepositRequestedEvent> jsonDeserializer = new JsonDeserializer<>(DepositRequestedEvent.class);
	    jsonDeserializer.addTrustedPackages("com.udemy.kafka.webapp.deposit.dto");
	    // This is crucial: ignore type headers from producer
	    jsonDeserializer.setUseTypeHeaders(false);

	    // Wrap with ErrorHandlingDeserializer
	    ErrorHandlingDeserializer<DepositRequestedEvent> errorHandlingDeserializer = new ErrorHandlingDeserializer<>(jsonDeserializer);

	    Map<String, Object> config = new HashMap<>();
	    config.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, environment.getProperty("spring.kafka.consumer.bootstrap-servers"));
	    config.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
	    config.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, errorHandlingDeserializer.getClass());
	    config.put(ConsumerConfig.GROUP_ID_CONFIG, environment.getProperty("spring.kafka.consumer.group-id"));
	    config.put(ConsumerConfig.ISOLATION_LEVEL_CONFIG, 
				environment.getProperty("spring.kafka.consumer.isolation-level", "READ_COMMITTED").toLowerCase());

	    return new DefaultKafkaConsumerFactory<String, DepositRequestedEvent>(config, new StringDeserializer(), errorHandlingDeserializer);
	}
	
	@Bean
	ConcurrentKafkaListenerContainerFactory<String, DepositRequestedEvent> kafkaListenerContainerFactory(
			ConsumerFactory<String, DepositRequestedEvent> consumerFactory, KafkaTemplate<String, DepositRequestedEvent> kafkaTemplate) {
		
		DefaultErrorHandler errorHandler = new DefaultErrorHandler(new DeadLetterPublishingRecoverer(kafkaTemplate), new FixedBackOff(5000, 3));
		errorHandler.addNotRetryableExceptions(NoTRetryableException.class);
		errorHandler.addRetryableExceptions(RetryableException.class);
		
		ConcurrentKafkaListenerContainerFactory<String, DepositRequestedEvent> factory = new ConcurrentKafkaListenerContainerFactory<>();
		factory.setConsumerFactory(consumerFactory);
		factory.setCommonErrorHandler(errorHandler);
		
		return factory;
	}
	
	@Bean
	KafkaTemplate<String, DepositRequestedEvent> kafkaTemplate(ProducerFactory<String, DepositRequestedEvent> producerFactory){
		return new KafkaTemplate<String, DepositRequestedEvent>(producerFactory);
	}

		config.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, environment.getProperty("spring.kafka.consumer.bootstrap-servers"));

Difference between @Transaction and kafkaTemplate.executeInTransaction
------------------------------------------------------------------------
@Transaction
------------
-> Main difference is a scope of a transaction. When you annotate a method with @Transactional, all business logic that you have in the method, is executed within the same transaction.
-> And this means that if exception is thrown anywhere in this method, then Kafka transaction will be rolled back and it doesn't matter where exception has been thrown in the method.
-> If exception is thrown even in the middle or end of the method, Whole Operation will be Rolled Back.

Local Transaction
-----------------
-> But if you use local transactions and then throw exception in the same method after the local transaction, then this local transaction will not roll back.
-> And this is because this exception did not take place within local transaction. It took place outside of the local transaction. Like below

public boolean transfer(TransferRestModel transferRestModel) {
  boolean returnValue = kafkaTemplate.executeInTransaction(t->{  //local transaction starts

	kafkaTemplate.send(withdrawMoneyTopic, withdrawalEvent);
	callRemoteServce();
    kafkaTemplate.send(depositMoneyTopic, depositEvent);
    
  })  //local transaction ends
  if(true) throw new Exception("Forcing Exception");  //Exception Occurs But Kafka messages are already committed
}

-> Even If you annotate transfer() with @Transactional, local transaction will not be rolled back cos this is a separate transaction. Below Example
@Transactional
public boolean transfer(TransferRestModel transferRestModel) {
	boolean returnValue = kafkaTemplate.executeInTransaction(t->{})  //local transaction
	if(true) throw new Exception("Forcing Exception");  //Exception Occurs But Kafka messages are already committed
}

Q) Apache Kafka and Database Transactions
A) From Now on, Will work with 2 Different Transaction Managers.
1) Kafka Transaction Manager:- Will Manage Kafka related Transactions. It wont manage DB related transactions.
2) JPA Transaction Manager:- Will Manage DB Operations related Transactions.

-> Within a transfer() method, if we want to publish msgs to Kafka topic and also want to perform Database Operations, we can use @Transactional.
-> Though we are performing Kafka and DB operations in the same method, they can still participate in the same transaction with @Transactional.
-> In this case, If exception takes place in the transfer() method, then both Kafka transaction and database transaction will roll back.

-> @Transactional("kafkaTransactionManager") will manage only Kafka related Transactions whereas
-> @Transactional("transactionManager") will manage both DB and Kafka related Transactions. Certainly All Transactions.

Q) Apache Kafka and Database Transactions Configurations By Java Code
A) Till now we have use @Transactional on top of transfer(), to send msgs on multiple Kafka Topics and if something goes wrong in between then whole transaction will be roll back.
-> We have configured 2 Different Transaction Managers. 1 is for Kafka and another is for DB related transactions.
-> We have configured both Transaction Manager by creating and using @Bean in KafkaTransferConsumerConfig class. Below is the syntax,
@Configuration
public class KafkaTransferConsumerConfig {
	
	@Bean("kafkaTransactionManager")
	KafkaTransactionManager<String, Object> kafkaTransactionManager(ProducerFactory<String, Object> producerFactory) {
		return new KafkaTransactionManager<>(producerFactory);
	}
	
	@Bean("transactionManager")
	JpaTransactionManager jpaTransactionManager(EntityManagerFactory entityManagerFactory) {
		return new JpaTransactionManager(entityManagerFactory);
	}
}

-> We have TransferEntity class to store data in DB
-> We are creating obj for TransferEntity and Copying data from TransferRestModel class which is actually we are using to Publish or Send Messages to Kafka Topic from Transfer Microservice by using HTTP Request.
-> By using below library, we are copying data from TransferRestModel to TransferEntity class. (Source to Destination Class)
BeanUtils.copyProperties(transferRestModel, transferEntity);  //(source, destination)

Senario-1
---------
-> Run Zookeeper and All 3 Brokers. Start Transfer, Deposit, Withdrawal and Mock Microservices.
-> Sequence is DB Save -> send Withdrawal Kafka Topic -> Call External HTTP Request.....
-> So once we publish data, we could see, Data have been stored in DB and msgs also been consumed by both Deposit and Withdrawal Consumers.

Senario-2
---------
-> Run Zookeeper and All 3 Brokers. Start Transfer, Deposit and Withdrawal Microservices. Stop Mock Microservice
-> Sequence is DB Save -> send Withdrawal Kafka Topic -> Call External HTTP Request which will throw Exception in this scenario
Expectation:-
-------------
-> When Exception occurs, Ideally it should rollback only DB Operations and Not Kafka Related Operations.
-> Because, we have used @Transactional("transactionManager") which is set to rollback only DB Operations and Not Kafka Operations.
Reality:-
---------
-> But since we have used "spring.kafka.producer.transaction-id-prefix" property, It makes your Kafka producer transactional.
-> And When you use a transactional Kafka producer inside any Spring-managed transaction (even if it’s a JPA transaction), Spring Kafka will synchronize the Kafka transaction with the outer transaction.
-> It will wait for Outside Transaction i.e DB Transaction(@Transactional("transactionManager")) to be committed, and this property is responsilble why both DB and Kafka are roll back.
-> And Since, Exception has Occurred and Outer or DB transaction is not committed, Kafka transaction is also not committed.
-> If the outer transaction is rolled back (because of an exception), the Kafka transaction is also rolled back.

Key Point
----------
-> "spring.kafka.producer.transaction-id-prefix" is what enables our Producer as Synchronized Transaction.
-> Without this property, Kafka Operations would not participate in the rollback.

Senario-3
---------
-> Run Zookeeper and All 3 Brokers. Start Transfer, Deposit and Withdrawal Microservices. Stop Mock Microservice
-> Sequence is DB Save -> send Withdrawal Kafka Topic -> Call External HTTP Request which will throw Exception in this scenario
-> Remove property "spring.kafka.producer.transaction-id-prefix" from Transfer Props file.
-> If we remove above Property then we also have to remove KafkaTransactionManager Bean from KafkaTransferConsumerConfig class, Like Below
@Bean("kafkaTransactionManager")
	KafkaTransactionManager<String, Object> kafkaTransactionManager(ProducerFactory<String, Object> producerFactory) {
	return new KafkaTransactionManager<>(producerFactory);
}
-> This is because, You cannot have a KafkaTransactionManager Bean without a transactional producer.
-> In our case, Transactional Producer is configured using property "spring.kafka.producer.transaction-id-prefix"
-> So, If you remove this property "spring.kafka.producer.transaction-id-prefix", you must also remove the KafkaTransactionManager bean.
-> If you want to keep the KafkaTransactionManager, you must configure the producer to be transactional.

-> This time, when Exception Occurs, Only DB Operations have been roll back and Not Kafka Operations.
-> Though Exception occurred, Withdrawal Kafka Consumer have read msgs cos we are only handled DB related Transactions and not Kafka Transactions.

 Saga Design Patterns
 --------------------
-> When 1 of the steps in Saga flow fails, microservices start performing compensating transactions.
-> Compensating transactions are operations that are microservices need to perform to undo the modifying changes that were done in our system, and compensating transactions are performed in a reverse order to Undo All the Operations
-> If any of the steps in the flow did not do any changes in our system, then we do not need to undo or no need to perform a compensating transaction.
